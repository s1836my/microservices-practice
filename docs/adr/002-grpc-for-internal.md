# ADR-002: サービス間同期通信に gRPC を採用する

## ステータス
Accepted

## コンテキスト

マイクロサービス間の同期 RPC 通信方式として以下の候補を検討した:

- **REST (HTTP/JSON)**: 実装が容易、デバッグしやすいが型安全性がない
- **gRPC (HTTP/2 + Protocol Buffers)**: 型安全、コード自動生成、高速
- **GraphQL**: 柔軟なクエリだがサービス間通信には over-engineering

## 決定

**gRPC + Protocol Buffers** を採用する。

採用理由:
1. **型安全なインターフェース定義**: `.proto` ファイルがサービス間の契約として機能し、型不整合をコンパイル時に検出
2. **コード自動生成**: `protoc` によりクライアント・サーバースタブを自動生成、ボイラープレートを排除
3. **HTTP/2 の恩恵**: ヘッダー圧縮・多重化によるレイテンシ削減
4. **インターセプター**: 認証・ログ・トレーシング・サーキットブレーカーを横断的に適用可能

## 結果

### メリット
- API Gateway が JWT から取得したユーザー情報をメタデータ経由で各サービスに伝播できる
- gRPC-go のインターセプターチェーンで OpenTelemetry トレーシングを透過的に追加
- スキーマ変更時の後方互換性を proto の optional フィールドで管理

### デメリット
- HTTP/1.1 クライアントから直接呼び出せない（API Gateway が REST-gRPC 変換を担う）
- デバッグに `grpcurl` 等のツールが必要

### 影響
- 全内部サービス間通信に適用
- `proto/` ディレクトリをリポジトリルートで一元管理
- `scripts/generate-proto.sh` による全サービスのスタブ一括生成
