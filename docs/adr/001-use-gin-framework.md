# ADR-001: API Gateway フレームワークに Gin を採用する

## ステータス
Accepted

## コンテキスト

API Gateway には HTTP リクエストを受け付け、gRPC クライアントへ転送する役割がある。
Go の HTTP フレームワーク選定において以下の候補を検討した:

- **net/http (標準ライブラリ)**: 依存なしだが、ルーティング・ミドルウェアの記述が煩雑
- **Gin**: 高速なゼロアロケーションルーター、豊富なミドルウェアエコシステム
- **Echo**: Gin と同等の性能、シンプルな API
- **Chi**: 軽量で標準ライブラリ互換

## 決定

**Gin (gin-gonic/gin)** を採用する。

理由:
1. **ルートグループとミドルウェアチェーン**: 認証が必要なルートと不要なルートを直感的に分離できる
2. **コミュニティ規模**: Go の Web フレームワーク中最大のスター数・採用実績
3. **Gin JWT ミドルウェア (appleboy/gin-jwt)**: JWT 検証の実績あるライブラリが存在
4. **パフォーマンス**: Radix ツリーによるルーティングで他フレームワークと同等以上の速度

## 結果

### メリット
- 充実したドキュメントとサンプルコード
- `r.Group()` + `Use()` によるクリーンなルーティング設計
- `c.ShouldBindJSON()` による自動バリデーション連携

### デメリット
- Echo や Chi に比べてわずかに opinionated
- 標準 net/http ハンドラとの互換レイヤーが必要な場合がある

### 影響
- API Gateway サービスのみに適用（他サービスは gRPC サーバーのため影響なし）
